# ECOCO AI 客服落地討論指南

本文件用於客服、營運、主管與系統維護者討論 ECOCO AI 客服如何從試營運走向正式使用。內容聚焦於客服流程、知識維護、LINE@ 串接前準備與內部責任分工，不要求讀者具備程式或資料庫背景。

## 1. 系統定位

ECOCO AI 客服是一套以 ECOCO 官方知識庫為依據的自動回覆系統。它不是讓 AI 自由回答，也不是取代客服人員做個案判斷，而是協助回答「可公開、可重複、已確認」的常見客服問題。

系統目前可支援：

- 網站 AI 客服問答。
- 客服後台知識維護。
- 知識缺口紀錄。
- 使用者回饋與負評追蹤。
- 主管報表。
- LINE Official Account Messaging API 串接準備。
- GitHub Actions 維運自動化。

## 2. 客服人員需要理解的範圍

| 項目 | 客服人員需要知道 | 不需要客服人員處理 |
| --- | --- | --- |
| AI 回覆來源 | AI 主要依據 ECOCO 知識庫回答 | 模型、embedding、資料庫 schema |
| 知識缺口 | AI 不確定或使用者負評時，會留下待處理項目 | 直接改程式或資料庫 |
| 知識維護 | 已確認的常見答案可補進後台 | 未確認個案不可寫成公開知識 |
| 高風險問題 | 補點、退款、帳號、客訴需保守回答並引導人工 | 讓 AI 承諾已補點、退款或已處理 |
| LINE@ 串接 | LINE 只是新的入口，仍共用同一套知識庫 | 在 LINE 後台另建一套 FAQ |

## 3. 上線前客服確認清單

正式擴大使用前，需由公司指定固定來源與負責窗口逐項確認下列內容。未確認項目不得寫成 AI 的正式回答。

| 類別 | 需確認內容 | 正式來源要求 | 負責窗口 | 無固定來源時處理原則 |
| --- | --- | --- | --- | --- |
| 點數規則 | 點數效期、點數查詢方式、點數未入帳處理流程 | 需公司指定固定來源 | 客服主管 / 會員系統窗口 | 不寫成 AI 可承諾答案，只引導人工查詢 |
| 兌換規則 | 優惠券兌換限制、已額滿或不可兌換時的標準說法 | 需公司指定固定來源 | 客服主管 / 商務或營運窗口 | 只回答一般原則，不承諾個案兌換結果 |
| 回收規則 | 寶特瓶、鋁罐、PP 杯、電池等允收品項與投遞規則 | 需公司指定固定來源 | 營運窗口 | 標記為待確認，不放入正式知識庫 |
| 機台異常 | 滿倉、故障、無法辨識、異物投入時的回報 SOP | 需公司指定固定來源 | 營運窗口 | 引導使用者提供站點、時間與照片，不承諾已派修 |
| 站點資訊 | 站點名稱、合作商家名稱、暫停服務或新站點資訊 | 需公司指定固定來源 | 營運窗口 | 只提供查詢方向，不回覆未確認的新站點或暫停資訊 |
| 客訴處理 | 補點、退款、賠償、帳號問題的人工流程 | 需公司指定固定來源 | 客服主管 | 一律引導人工處理，不讓 AI 做最終判定 |
| 品牌語氣 | LINE@ 與網站客服的稱呼、語氣、emoji 使用方式 | 需公司指定固定來源 | 行銷 / 客服主管 | 使用保守、中性、禮貌語氣，避免過度口語或未核准表述 |
| 人工表單 | 客服表單連結、需收集欄位、轉人工條件 | 需公司指定固定來源 | 客服主管 / 維護者 | 不放未確認連結；改請使用者依官方客服管道聯繫 |

## 4. 試營運建議流程

建議不要一次接上所有正式流量。較安全的落地方式如下：

1. 由客服與營運先確認核心知識庫。
2. 用網站前台測試 20 至 30 題常見問題。
3. 特別測試高風險問題，例如點數未入帳、退款、優惠券無法兌換、機台異常與帳號問題。
4. 確認 AI 不會做未經授權的承諾。
5. 讓客服人員熟悉後台的知識缺口、負評與知識維護流程。
6. 再接 LINE@ 測試 Channel 或小範圍測試使用者。
7. 每週檢查報表、知識缺口與負評，持續調整知識庫。

## 5. 日常維護方式

### 客服日常

客服人員每日或每週固定查看：

- AI 不確定問題。
- 使用者負評。
- 高風險問題是否有錯誤承諾。
- 常見重複問題是否需要補知識。

### 知識更新

可補進知識庫的內容：

- 已由主管或營運確認的 FAQ。
- 已確認的標準 SOP。
- 可公開給所有使用者的規則。
- 不涉及個資與個案判斷的通用回答。

不應補進知識庫的內容：

- 單一會員個案結果。
- 內部責任歸屬。
- 未確認的新活動或新站點。
- 私人手機、email、會員資料。
- 尚未授權公開的合作資訊。

## 6. 重大更新與交接

後台修改會立即寫入 PostgreSQL，但不會自動寫回 GitHub。重大更新、交接或正式備份前，請執行下列流程：

1. 從後台下載 JSON。
2. 人工確認內容正確且無個資。
3. 覆蓋 `data/ecoco-knowledge-import.json`。
4. 由維護者 commit / push。
5. 確認 GitHub Actions 與 Render 部署狀態。

## 7. 角色分工

| 角色 | 負責事項 |
| --- | --- |
| 客服人員 | 查看知識缺口、回報錯誤回答、補充已確認常見問題 |
| 客服主管 | 決定高風險問題口徑、確認不可承諾內容、排定知識缺口優先順序 |
| 營運窗口 | 確認站點、機台、回收品項、活動與合作資訊 |
| 技術維護者 | 維護部署、API key、資料庫、GitHub、LINE Webhook 與備份流程 |
| 專案負責人 | 整理交接文件、安排試營運、彙整客服與主管回饋 |

## 8. 會議建議議程

與客服人員討論落地時，建議使用以下議程：

1. 說明 AI 客服定位：回答常見問題，不做個案裁決。
2. 展示前台提問與後台知識缺口。
3. 確認高風險問題處理邊界。
4. 確認 LINE@ 是否採 Messaging API Webhook 串接。
5. 確認客服人員需要維護哪些資料。
6. 確認正式上線前還缺哪些公司資源。
7. 指定知識庫與高風險規則的審核窗口。

## 9. 判斷是否可擴大上線

符合下列條件後，才建議擴大至正式 LINE@ 流量：

- 公司正式 API key、Render、PostgreSQL 與 GitHub 權限已確認。
- LINE Developers 權限與 Webhook 設定已完成。
- 客服與主管已確認核心規則。
- 公司已指定知識庫正式來源、負責窗口與無固定來源時的處理原則。
- 高風險問題測試通過。
- 知識缺口與負評流程有人負責。
- 每週備份、健檢與監控流程可正常執行。
