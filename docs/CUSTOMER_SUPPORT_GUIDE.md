# ECOCO AI 客服後台操作指南

本文件提供客服與營運人員使用 ECOCO AI 客服後台的標準作業流程。內容聚焦於日常查看、知識缺口處理、知識庫維護與回覆風險控管，不要求使用者具備 SQL、Git 或部署經驗。

## 1. 使用對象與責任

| 角色 | 主要責任 | 不需要處理 |
| --- | --- | --- |
| 客服人員 | 查看 AI 回覆狀況、標記問題、補充已確認的常見答案 | 部署、資料庫 schema、API key |
| 營運人員 | 協助確認站點、機台、活動與合作方資訊 | 修改程式碼、調整模型 |
| 客服主管 | 監督高風險回覆、確認知識缺口優先順序 | 直接操作資料庫 |
| 系統維護者 | 維護知識庫、匯出 JSON、處理部署與資料同步 | 代替客服判定個案結果 |

## 2. 後台頁面分區

後台分成三個主要分區，避免所有資料擠在同一個頁面。

| 分區 | 使用情境 | 建議使用者 |
| --- | --- | --- |
| 客服日常 | 查看 AI 不確定問題、使用者評分與對話紀錄 | 客服、營運 |
| 知識維護 | 新增、修改、封存、恢復知識分類 | 維護者、指定客服 |
| 主管報表 | 查看客服量、問題分類、知識缺口與報表內容 | 主管、營運、維護者 |

## 3. 每日作業流程

建議每日依照下列順序檢查：

1. 進入「客服日常」，查看是否有 AI 不確定問題。
2. 查看使用者負評或「需改善」回饋。
3. 檢查近期對話中是否有重複出現的問題。
4. 若問題已有正確答案，到「知識維護」補上或修改知識。
5. 若問題尚未確認，保留待辦並註記需要哪個內部窗口確認。
6. 補完資料後，回到前台用相似問法測試 AI 回覆。
7. 若回覆正確，將相關待辦標記為已處理或刪除待辦項目。

## 4. 知識缺口處理

知識缺口代表 AI 目前沒有足夠依據可以穩定回答。它不是錯誤清單，而是「需要人工補知識或判斷是否不用處理」的待辦清單。

| 狀態 | 判斷方式 | 處理方式 |
| --- | --- | --- |
| 待確認 | 還不知道標準答案 | 保留，交由主管或相關窗口確認 |
| 已補資料 | 已將標準答案補進知識庫 | 前台測試後刪除待辦 |
| 需人工處理 | 涉及個案、帳號、補點、退款、客訴 | 引導客服表單或人工流程，不寫成公開知識 |
| 不需處理 | 非 ECOCO 業務、無效訊息、測試訊息 | 可刪除待辦，避免列表膨脹 |

處理原則：

- 只把可公開、可重複使用的內容寫進知識庫。
- 個案判定、內部責任歸屬、會員資料與補償結果不可寫成 AI 公開知識。
- 若答案尚未經內部確認，不要為了讓 AI 能回答而硬寫內容。

## 5. 知識庫新增與修改

### 5.1 新增前先搜尋

新增知識前，請先搜尋既有分類，避免重複建立相似分類。

例：

| 想新增的內容 | 建議先搜尋 |
| --- | --- |
| 點數沒入帳 | 點數、入帳、回收點數 |
| 優惠券不能換 | 優惠券、兌換、合作商家 |
| 機台滿倉 | 滿倉、機台、清潔、維修 |
| 新竹設點 | 站點、設點、地點 |

若已有相近分類，優先修改既有分類，不要新增重複分類。

### 5.2 建議填寫格式

建議每筆知識包含以下段落：

```text
適用情境：
使用者通常會怎麼問。

標準回覆：
客服可以公開回覆的正式內容。

需要收集資訊：
若需要人工查詢，應請使用者提供哪些資料。

不可承諾：
例如不可承諾補點、退款、已處理或工程師已到場。

內部備註：
只給客服或維護者看的提醒。
```

### 5.3 封存與刪除

若知識過期、不確定或暫時不應讓 AI 使用，請優先使用「封存」。

封存後：

- 資料仍留在 PostgreSQL。
- AI 不會使用該筆知識回答。
- 後續仍可恢復。

只有在確認該內容不需要保留時，才考慮刪除待辦或清理資料。

## 6. 高風險問題處理

以下問題必須使用保守話術，不得讓 AI 做未確認承諾。

| 類型 | 可以回覆 | 不可以回覆 |
| --- | --- | --- |
| 點數未入帳 | 請使用者提供註冊電話、回收時間、站點、截圖 | 已補點、一定會補、已完成查詢 |
| 優惠券無法兌換 | 說明可能原因，引導客服表單 | 保證可以兌換、承諾補償 |
| 機台異常 | 收集站點、時間、照片或截圖 | 判定責任、承諾維修完成 |
| 帳號問題 | 引導客服表單與必要資訊 | 查詢或公開個資 |
| 客訴爭議 | 表達理解，轉人工或客服表單 | 代表公司做最終判定 |

客服檢查 AI 回覆時，若看到過度肯定、未確認承諾或錯誤資訊，應立即補充知識庫的正確說法，並將相關待辦標記為需人工處理。

## 7. 報表使用方式

主管報表可用於週報、月報與營運觀察。

建議觀察：

- 客服案件數
- 使用者訊息數
- AI 回覆數
- 知識缺口數
- 問題分類分布
- 已處理與待處理項目
- 使用者評分與負評案例

注意：報表數字可作為營運觀察，不應直接宣稱節省成本或創造效益。若需要成本效益分析，需另行使用主管確認的估算公式。

## 8. JSON 匯出與版本備份

「下載 JSON」只會把目前 PostgreSQL 裡的知識庫匯出成檔案，不會自動更新 GitHub。

以下情況應下載 JSON 並交給維護者備份：

- 大量更新知識後
- 交接前
- 主管要求保存正式版本時
- 重大活動、合作方或點數規則更新後

維護者需人工檢查匯出內容，再覆蓋 `data/ecoco-knowledge-import.json` 並 commit / push。

## 9. 交接前檢查清單

交接前請確認：

- 後台可以登入。
- 最近知識缺口已處理或標記狀態。
- 高風險問題沒有錯誤承諾。
- 重大知識更新已下載 JSON。
- 重要資料已由內部窗口確認。
- GitHub 上的正式資料與文件已更新。
- Render、PostgreSQL 與 API key 權限由公司帳號控管。

## 10. 對客服人員的原則

AI 客服不是取代客服判斷，而是協助回答可公開、可重複、已確認的常見問題。遇到個案、個資、補償、帳號、退款或客訴判定時，應回到人工流程，避免讓 AI 做超出權限的承諾。
