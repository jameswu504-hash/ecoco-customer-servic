# ECOCO AI 客服：客服人員操作指南

這份文件是給一般客服人員看的版本，不需要懂 SQL、Git、Render 或 RAG。

客服人員只需要理解三件事：

1. AI 客服怎麼回答。
2. 後台哪些地方要每天看。
3. 看到 AI 答不出來或答錯時，要怎麼處理。

## 一、這套 AI 客服在做什麼

ECOCO AI 客服會根據已整理好的 ECOCO 知識庫回答使用者問題。

它不是自己亂猜，也不是直接連到會員後台查個資。它主要回答：

- App 使用方式。
- 回收規則。
- 點數規則。
- 優惠券與兌換說明。
- 站點與機台常見問題。
- 一般客服 SOP。

如果問題涉及個資、補點、退款、帳號異常、客訴爭議，AI 應該保守回答，請使用者提供必要資訊或引導客服表單。

## 二、客服人員每天要看哪裡

請打開後台：

```text
https://ecoco-customer-servic.onrender.com/dashboard.html
```

後台主要看四個區塊：

| 區塊 | 客服人員要做什麼 |
| --- | --- |
| 營運報表 | 看本週/本月客服量、問題分類、知識缺口數 |
| 每次對話詳細紀錄 | 看使用者問什麼、AI 怎麼回 |
| 知識缺口列表 | 看 AI 無法完整回答的問題 |
| 知識庫管理 | 補充或修改 AI 可以使用的知識 |

新版後台最上方有「客服日常工作台」，客服人員可以照 1 到 4 的卡片操作：

| 工作台卡片 | 用途 |
| --- | --- |
| 先看營運報表 | 了解本週/本月客服量、分類、知識缺口 |
| 處理 AI 不確定問題 | 看待辦問題，判斷要補知識或刪除 |
| 補充客服知識 | 跳到知識庫管理，新增或修改分類內容 |
| 查詢對話紀錄 | 回看使用者原話與 AI 回覆 |

後台也分成三個層級，避免所有資料都擠在同一頁：

| 分頁 | 誰主要使用 | 用途 |
| --- | --- | --- |
| 客服日常 | 一般客服 | 看 AI 不確定問題、評分、對話紀錄 |
| 知識維護 | 維護者/熟悉資料的人 | 補知識庫、下載 JSON 備份、查看資料來源 |
| 主管報表 | 主管/總經理室 | 看客服量、問題分類、週報/月報 |

## 三、每天建議流程

建議每天或每週固定做一次：

```text
登入後台
  ↓
看營運報表
  ↓
看知識缺口
  ↓
看負評或奇怪對話
  ↓
判斷是否需要補知識
  ↓
需要補：到知識庫管理新增/修改
  ↓
不需要補：刪除或標記不需處理
  ↓
用前台測一次同類問題
```

## 四、知識缺口是什麼

知識缺口就是 AI 發現「目前知識庫沒有明確答案」的問題。

例如：

- 使用者問新的活動，但知識庫還沒有。
- 使用者問某個合作商家兌換問題，但資料不完整。
- 使用者問站點是否要新增，AI 沒有官方答案。
- 使用者問個人點數或帳號狀態，AI 不能直接查。

知識缺口不是錯誤，而是提醒客服：

```text
這個問題可能要人工確認，或補進知識庫。
```

## 五、知識缺口怎麼處理

### 可以補進知識庫

如果這個問題是一般 FAQ，而且答案已經確認，可以補進知識庫。

例子：

- 「點數多久到期？」
- 「寶特瓶要不要壓扁？」
- 「優惠券不能兌換怎麼辦？」
- 「機台紅燈代表什麼？」

處理方式：

1. 到後台「知識庫管理」。
2. 找到相關分類。
3. 補上正確答案。
4. 儲存。
5. 回到前台測試同類問題。
6. 確認 AI 回得正確後，回知識缺口列表刪除該筆。

新增分類時，請先在分類名稱輸入關鍵字，例如「點數」或「兌換」。後台會提示已存在的相近分類；如果有相近分類，優先開啟原分類編輯，不要重複新增。

### 不應該補進知識庫

如果問題涉及個資或單一會員狀態，不應補成通用知識。

例子：

- 「我的點數為什麼不見？」
- 「我帳號被誰登入？」
- 「幫我補點。」
- 「幫我退款。」
- 「我的優惠券是不是已核銷？」

處理方式：

1. 不要讓 AI 直接承諾處理結果。
2. 引導使用者填客服表單或 App 聯絡我們。
3. 請使用者提供必要資訊，例如註冊手機、時間、站點、截圖。
4. 若後台有狀態選項，標成「需人工處理」。

## 六、哪些問題 AI 不能直接承諾

遇到以下問題，AI 不應該說「已幫你處理」、「已補點」、「已退款」、「已通知工程師」：

| 類型 | AI 應該怎麼做 |
| --- | --- |
| 點數未入帳 | 收集註冊手機、回收時間、站點、截圖 |
| 優惠券不能用 | 收集券名稱、錯誤畫面、使用時間 |
| 帳號問題 | 引導 App 聯絡我們或客服表單 |
| 補點/退款/賠償 | 不承諾結果，交由人工確認 |
| 機台故障 | 收集站點、時間、機台狀況 |
| 客訴 | 先同理，再引導人工流程 |

客服人員可以記住一句話：

```text
AI 可以回答規則，但不能替公司承諾個案處理結果。
```

## 七、怎麼新增知識才比較安全

新增知識時，建議用這個格式：

```text
### 問題標題

適用情境：
- 這段答案適合什麼問題。

建議回覆：
- 給使用者看的正式回答。

需要收集資訊：
- 註冊手機
- 發生時間
- 站點名稱
- 截圖

注意事項：
- 不承諾補點、退款或已處理。
```

## 八、常見分類建議

| 問題 | 建議放入分類 |
| --- | --- |
| 點數多久到期 | 點數問題 |
| 點數沒入帳 | 點數問題 / 高風險保守話術 |
| 優惠券不能用 | 優惠券 / 兌換 |
| 漢堡王為何不能兌換 | 合作商家 / 優惠券 |
| 機台滿倉 | 機台 / 維修 |
| 機台旁邊都是垃圾 | 現場環境 / 清潔 |
| 可以在新竹設點嗎 | 站點建議 |
| 收不到驗證碼 | APP / 帳號 |
| 如何註冊會員 | APP / 帳號 |

## 九、如何判斷 AI 回答好不好

看 AI 回覆時，客服人員可以用這 5 個問題檢查：

1. 有沒有回答使用者真正問的問題？
2. 有沒有引用錯誤或過期資訊？
3. 有沒有承諾補點、退款、人工已處理？
4. 有沒有請使用者提供必要資訊？
5. 語氣是否符合 ECOCO：親切、專業、清楚？

如果第 2 或第 3 點有問題，應優先修正知識庫或回覆規則。

## 十、客服人員不需要懂的東西

客服人員不需要操作：

- GitHub commit / push。
- Render 部署。
- SQL 指令。
- `.env`。
- API key。
- Claude 模型設定。

這些由維護者或工程人員處理。

客服人員只需要會：

- 看後台。
- 看知識缺口。
- 補一般 FAQ。
- 把高風險個案交人工。
- 回報 AI 答錯或資料不足。

## 十一、回報問題給維護者時要附什麼

如果發現 AI 答錯，請附：

1. 使用者原問題。
2. AI 的錯誤回答。
3. 正確答案或希望改成什麼。
4. 這是一般 FAQ 還是個案問題。
5. 是否需要主管確認。

範例：

```text
問題：使用者問「點數效期多久？」
AI 回答：點數終生有效。
正確答案：點數效期為 12 個月，以官方公告規則為準。
類型：一般 FAQ，需立即修正。
```

## 十二、一句話總結

```text
客服人員負責判斷 AI 回覆是否符合客服現場；系統負責記錄問題與提供知識庫；高風險個案仍要交由人工確認。
```
