# ECOCO AI 客服後台操作指南

本文件提供 ECOCO 客服與營運人員使用 AI 客服後台的標準作業方式。內容聚焦於日常查看、問題判斷、知識補充與回覆風險控管；不需要具備 SQL、Git 或部署經驗。

## 1. 文件適用範圍

| 項目 | 說明 |
| --- | --- |
| 適用對象 | 客服人員、營運人員、知識庫維護協助者 |
| 使用系統 | ECOCO AI 客服前台與內部後台 |
| 主要任務 | 查看對話、處理知識缺口、維護客服知識、產出營運報表 |
| 不包含 | 程式部署、資料庫結構調整、API key 管理 |

## 2. 系統用途

ECOCO AI 客服會依照已整理的 ECOCO 知識庫，自動回答使用者常見問題。系統目前適合處理：

- App 使用方式
- 回收規則與機台操作
- 點數規則與優惠券兌換說明
- 站點、機台、滿倉、清潔等常見問題
- 一般客服引導與客服表單提醒

AI 不應直接處理或承諾以下事項：

- 補點、退款、賠償
- 帳號、會員資料、個資查詢
- 客訴責任判定
- 合作方未確認公告
- 內部尚未確認的新站點或新活動資訊

遇到上述情境時，客服人員應確認 AI 是否使用保守話術，並引導使用者提供必要資料或填寫客服表單。

## 3. 後台分區

後台分為三個使用層級，避免所有資料集中在同一頁造成維護困難。

| 分區 | 使用者 | 用途 |
| --- | --- | --- |
| 客服日常 | 客服人員 | 查看 AI 不確定問題、使用者評分、對話紀錄 |
| 知識維護 | 知識庫維護者 | 新增、修改、封存或恢復知識分類 |
| 主管報表 | 主管、營運、總經理室 | 查看客服量、問題分類、處理狀態與週報/月報 |

## 4. 每日作業流程

建議客服人員每日依照下列順序檢查：

1. 查看「主管報表」確認近期客服量與主要問題類型。
2. 查看「客服日常」中的 AI 不確定問題。
3. 檢查低評分或負面回饋，確認 AI 是否回答錯誤或過度承諾。
4. 若缺少資料，整理問題並請內部窗口確認正確答案。
5. 到「知識維護」新增或修改知識。
6. 若某筆知識不應再給 AI 使用，使用「封存」，不要直接刪除。
7. 更新後回到前台，以類似問題測試 AI 回覆是否改善。

## 5. 知識缺口處理原則

知識缺口代表 AI 判斷目前資料不足，或回答中出現「需要客服確認」的情境。它不是錯誤清單，而是後續補資料的待辦清單。

| 狀態 | 使用時機 | 後續動作 |
| --- | --- | --- |
| 待確認 | 還不知道正確答案 | 請主管、營運或相關窗口確認 |
| 已補資料 | 已將正確內容補進知識庫 | 用前台測試同類問題 |
| 需人工處理 | 涉及帳號、補點、退款、客訴、個資 | 引導客服表單或人工處理 |
| 不需處理 | 非 ECOCO 業務、惡意輸入、無需回答 | 可留紀錄後移除待辦 |

處理知識缺口時，請優先補充「可公開回答、可重複使用」的內容。個案處理結果、個資、內部判斷不應寫成公開知識。

## 6. 知識庫維護方式

### 6.1 新增知識

新增知識前，先在左側搜尋既有分類，避免重複建立相似分類。例如要新增點數規則，請先搜尋「點數」或「兌換」。

建議內容格式：

```text
適用情境：
使用者可能會怎麼問。

標準回覆方向：
客服可公開回覆的正式說法。

需收集資訊：
若需要人工查詢，應請使用者提供哪些資料。

不得承諾：
例如不得承諾已補點、已退款、已完成處理。

內部備註：
只給客服或維護者看的提醒。
```

### 6.2 修改知識

若只是修正用字、補充說明或更新活動狀態，可直接在既有分類修改。修改後請用前台測試至少一個相關問題。

### 6.3 封存知識

若某筆知識已過期、不確定或不應再被 AI 使用，請使用「封存」。封存後資料仍保留在 PostgreSQL，但不會進入 AI 檢索與回答。

封存適用於：

- 過期活動
- 舊合作方說法
- 尚待內部確認的內容
- 已被新版本取代的回覆

### 6.4 下載 JSON

「下載 JSON」只會把目前 PostgreSQL 的有效知識匯出成檔案，不會自動寫回 GitHub。

請在以下情境下載 JSON 並交給維護者放回 Git：

- 大量修改知識後
- 交接前
- 重要版本備份前
- 主管要求保留正式版本時

## 7. 回覆風險控管

客服人員檢查 AI 回覆時，請特別注意下列高風險情境。

| 類型 | AI 可做 | AI 不可做 |
| --- | --- | --- |
| 點數未入帳 | 請使用者提供註冊電話、時間、地點 | 承諾補點或說已處理 |
| 優惠券無法兌換 | 說明可能原因與客服表單 | 承諾可兌換或補償 |
| 機台異常 | 收集站點、時間、照片或截圖 | 判定責任或承諾維修完成 |
| 帳號問題 | 引導客服表單與必要資料 | 查詢、公開或推測個資 |
| 客訴爭議 | 表達理解並轉人工 | 代替公司做最終判定 |

如果 AI 回覆過度肯定、承諾補償、或使用未確認資訊，請立即將問題標記為需人工處理，並更新知識庫的保守說法。

## 8. 營運報表使用方式

主管報表可用來整理週報或月報。建議觀察：

- 客服案件數
- 使用者訊息數
- AI 回覆數
- 知識缺口數
- 問題分類分布
- 已處理與待處理項目
- 使用者評分與負評案例

報表中的數字應作為營運觀察，不應直接宣稱節省成本或創造效益；若需要成本效益分析，應另外使用經主管確認的估算公式。

## 9. 交接提醒

交接時請確認：

- 後台可以登入。
- 最近知識缺口已處理或標記狀態。
- 重大修改已下載 JSON 備份。
- 重要知識已請內部窗口確認。
- GitHub 上的正式文件與資料有同步更新。
- Render 與 PostgreSQL 權限由公司帳號控管。

日常客服只需要維護後台資料；正式版本整理、GitHub 更新與部署，應由維護者或指定窗口處理。
