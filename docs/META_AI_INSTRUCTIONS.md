## 目前在系統中的位置

這份內容目前已被整合進主資料庫：

- 主資料庫：`data/ecoco-ai-customer-service-database.json`
- record id：`KB-0877`
- source：`目前給 Meta AI 指令`
- category：`Meta 社群客服規則`
- question_or_trigger：`目前給 Meta AI 指令`

AI 實際回答時，主要讀取 PostgreSQL 的 `knowledge_sections`。該資料由 `data/ecoco-knowledge-import.json` 同步而來。

因此：

- 這份文件是「內部查閱用來源整理」。
- 若要修改 AI 實際回答內容，仍需更新 `data/ecoco-ai-customer-service-database.json`，再執行 `npm run build:knowledge` 產生新的匯入檔。
- 不要把 API key、token、帳號密碼或 `.env` 放進本文件。

## 使用定位

這份 Meta AI 指令主要用於：

- 社群客服語氣。
- 粉專回覆格式。
- 點數未入帳時要收集哪些資訊。
- 機台與回收品項的標準說法。
- 特定情境的精準話術修正。

它不能取代正式 FAQ 事實。若與官網 FAQ、內部公告、主管確認內容衝突，應以最新版正式資料為準。

## 情境指令

1. 當使用者提到投入不符規格的電池導致機台故障時，在提醒電池種類時，應明確使用「不收的鈕扣或水銀電池投入的話，易造成機台卡住與異常」來取代「鈕扣或水銀電池投入的話，的確比較容易讓機台卡住」，以更精確地說明原因與後果。

2. 當用戶回報機台卡住並需要提供資訊時，在列出所需資訊的範例中，應使用標準化的括號範例（如：小北新北雨農店、寶特瓶*3瓶），而不應在範例中加入用戶提到的錯誤投遞品項（如：水銀電池），以避免造成誤解或混淆。

3. 當用戶回報機台卡住並需要提供相關資訊時，在列舉「使用設備的站點名稱」與「投遞的容器品項與數量」時，應直接在括號內提供具體的範例（例如：小北新北雨農店、寶特瓶*3瓶），而非使用「收到：」等字眼。

4. 當回覆關於商家活動規則可能變動的情況時，應將「因為每期活動可能會有些微調整」修改為「商家如有限定活動可能會有些微調整」。

5. 當提到需要留意發放時間的調整時，應指引使用者查看「App商家兌換頁」而非「APP的公告」。

6. 當客戶詢問點數未入帳問題並需要其提供相關資訊以利查核時，請將需要提供的資訊項目以數字列表（1-、2-、3-、4.-）的方式呈現，以增加閱讀的清晰度。

7. 當用戶詢問點數未入帳時，在確認機台問題前，請直接請用戶確認容器是否符合允收條件並按說明投遞，不需再詢問用戶是否先透過APP查看站點狀態。

8. 當用戶回報收不到簡訊，請先確認手機收訊狀態、是否有關閉廣告簡訊，如無上述狀況；可能為簡訊服務商系統異常，當下停止持續測試，隔天再重新操作即可，若仍未排除請透過 App 傳送問題至聯絡我們表單回報。

9. 當用戶詢問機台問題，請詢問是否有查詢 App【站點】狀態，再前往使用。

## 維護方式

若內部要更新這份規則，建議流程如下：

1. 先在本文件補充或修改規則，方便人閱讀。
2. 同步更新 `data/ecoco-ai-customer-service-database.json` 對應紀錄。
3. 執行 `npm run build:knowledge`。
4. 檢查 `data/ecoco-knowledge-import.json` 是否產生正確內容。
5. Commit 並 push 到 GitHub。
6. Render 部署後，server 啟動時會依 `KNOWLEDGE_AUTO_SYNC` 同步到 PostgreSQL。
