# ECOCO AI 客服：LLM Wiki、Rule 與模型應用策略

本文件整理 mentor 提到的三個方向：LLM Wiki、Gemma/Llama/Ollama 本地模型，以及 Rule。目標不是立刻把系統改得很複雜，而是讓 ECOCO AI 客服有一個清楚、可維護、可交接的 AI 知識與模型使用方式。

## 一、結論

目前正式客服回覆建議維持 Claude Sonnet 作為主要模型。原因是客服回答需要穩定、語氣一致、能遵守規則，且目前系統已經接好 Claude、PostgreSQL、RAG 檢索與風險控管。

Gemma、Llama、Ollama 可以導入，但建議先用在內部輔助工作，例如資料整理、知識分類、重複資料檢查、回覆草稿品質檢查，不建議直接取代正式客服回覆模型。

Rule 是這個專案最重要的安全層之一。它不是多餘的規定，而是用來限制 AI 不亂承諾、不亂補點、不亂說已處理、不把高風險問題當一般 FAQ 回答。

## 二、LLM Wiki 是什麼

LLM Wiki 可以理解成「給 AI 和維護人員看的客服知識維基」。它不是單一檔案，而是一套知識管理方式。

在本專案中，LLM Wiki 應該包含：

| 類型 | 用途 | 對應檔案或資料表 |
| --- | --- | --- |
| 正式客服知識 | AI 實際回答會參考的內容 | `data/ecoco-knowledge-import.json`、PostgreSQL `knowledge_sections` |
| 回答規則 | 告訴 AI 哪些能答、哪些要保守、哪些要轉人工 | `data/ecoco-response-policies.json` |
| 品牌語氣 | ECOCO 的稱呼、語氣、口吻與禁用說法 | `docs\META_AI_INSTRUCTIONS.md`、知識庫中的品牌規則 |
| 維護流程 | 內部人員如何新增、修改、封存、匯出知識 | `docs\MAINTENANCE_GUIDE.md`、`docs\OPERATIONS_HANDOFF_GUIDE.md` |
| 技術交接 | 工程端如何部署、匯入、更新資料 | `docs\DEPLOYMENT_RUNBOOK.md`、`docs\DATA_DICTIONARY.md` |

換句話說，LLM Wiki 不是要再多開一個新系統，而是把現有的 Git JSON、PostgreSQL、Rule、文件整理成一套「誰都知道去哪裡看、怎麼改、怎麼上線」的管理方式。

## 三、目前專案已經具備的部分

目前已完成：

- 前台客服頁面：使用者可以直接提問。
- 後台知識庫管理：內部人員可以新增、修改、封存知識。
- PostgreSQL 知識表：正式線上服務讀取 `knowledge_sections` 與 `knowledge_chunks`。
- Git JSON 正式版本：`data/ecoco-knowledge-import.json` 保存可追蹤版本。
- RAG 檢索：優先使用 pgvector 語意檢索，沒有 embedding 設定時會退回關鍵字檢索。
- Rule / Guardrail：高風險資料會透過 `risk_level` 觸發保守回覆規則。
- 知識缺口：AI 不確定或無法回答時，會留下待補資料。

目前需要繼續補強：

- 把內部知識來源命名更一致，避免同一類資料有多個相似分類。
- 建立固定維護節奏，例如每週檢查知識缺口、每月匯出 JSON 回 Git。
- 將 Rule 更細分成可讀、可審核、可測試的政策。
- 將本地模型用在資料整理與測試，而不是直接接到正式客服。

## 四、Gemma、Llama、Ollama 可以怎麼用

### 1. Gemma / Llama 是什麼

Gemma 是 Google 的開源模型系列，Llama 是 Meta 的開源模型系列。常見的 8B、16B 是模型大小，數字越大通常能力越好，但需要的電腦資源也越高。

### 2. Ollama 是什麼

Ollama 是在本機跑開源模型的工具。它可以讓你不用每次都呼叫雲端 API，就能在自己的電腦上測試 Gemma、Llama 等模型。

### 3. 在 ECOCO 專案中的建議用途

| 用途 | 是否建議 | 原因 |
| --- | --- | --- |
| 正式客服自動回覆 | 暫不建議 | 品質、穩定性、規則遵守能力需要更多測試 |
| 知識分類建議 | 建議 | 可以協助把新知識歸到正確分類 |
| 重複知識檢查 | 建議 | 可以找出相似問法與重複內容 |
| 回覆品質檢查 | 建議 | 可以當第二層檢查，例如是否過度承諾 |
| 內部文件摘要 | 建議 | 可協助整理週報、月報、維護紀錄 |
| 成本較低的測試環境 | 可試 | 適合用來做離線實驗，不影響正式服務 |

### 4. 不建議現在直接做的事

- 不建議直接把 Claude 拔掉改用 Gemma 或 Llama。
- 不建議讓本地模型直接回覆真實客戶。
- 不建議在還沒有評測資料前，就宣稱哪個模型比較好。

## 五、Rule 是什麼，為什麼要學

Rule 是 AI 客服的「回答邊界」。它決定 AI 在不同情境下可以說什麼、不可以說什麼。

ECOCO 客服特別需要 Rule，因為有很多問題不能隨便回答：

- 點數未入帳：不能直接承諾補點。
- 優惠券不能兌換：不能直接承諾補償或兌換成功。
- 機台異常：不能說工程師已處理，除非資料來源確認。
- 帳號與個資：不能要求使用者公開提供完整個資。
- 客訴案件：不能把 AI 回覆包裝成人工已審核結果。

本專案目前的 Rule 來源：

- `data/ecoco-response-policies.json`
- `knowledge_chunks.risk_level`
- 系統提示詞中的高風險回覆規則
- 知識庫內容中的「需要收集資訊」與「不能承諾事項」

## 六、建議的導入順序

### Phase 1：整理 LLM Wiki

把所有知識來源、規則、更新流程整理清楚，讓客服、實習生、工程師都知道去哪裡看。

交付物：

- 本文件
- 維護手冊
- 資料來源清單
- 知識分類規範

### Phase 2：強化 Rule

把「高風險」與「可自動回答」定義得更清楚。

交付物：

- 高風險問題清單
- 可自動回答問題清單
- 需轉人工問題清單
- 回覆禁語與保守話術範本

### Phase 3：導入本地模型作為內部輔助

先讓 Gemma / Llama / Ollama 協助整理資料，不接正式客服。

可做功能：

- 新增知識時，自動建議分類。
- 找出重複知識。
- 檢查回答是否有過度承諾。
- 協助產生內部週報摘要。

### Phase 4：模型評測與分流

建立測試題庫，比較 Claude、Gemma、Llama 的回答品質。

可評估指標：

- 答案正確率
- 是否遵守 Rule
- 語氣是否符合 ECOCO
- 是否能處理知識缺口
- 成本與速度

### Phase 5：正式模型路由

如果未來資料證明本地模型穩定，才考慮分流：

- 一般 FAQ：成本較低模型
- 高風險問題：Claude Sonnet
- 內部分析與報表：可用本地模型或批次模型

## 七、維護原則

1. 正式客服回答以穩定與正確為優先，不以炫技為優先。
2. 開源模型先用在內部輔助，不直接面向客戶。
3. Rule 要比模型更重要，因為客服錯誤承諾會造成營運風險。
4. PostgreSQL 是線上運作資料庫，Git JSON 是正式版本備份與交接來源。
5. 大改知識庫或交接前，必須從後台下載 JSON，人工覆蓋回 `data/ecoco-knowledge-import.json`，再 commit / push。

## 八、給主管或 mentor 的說明版本

本專案可以套用 LLM Wiki 的概念，但不需要另外重做一套 wiki 系統。現階段做法是把 Git JSON、PostgreSQL 知識庫、response policies、維護文件整合成一套可追蹤的 AI 知識管理流程。

Gemma、Llama、Ollama 可以作為內部輔助模型，用來做資料分類、重複檢查、回覆品質檢查與文件摘要；正式客服回覆仍建議先維持 Claude Sonnet，以確保穩定性、品牌語氣與高風險規則遵守。

Rule 是 AI 客服落地的必要安全設計，能限制模型不要對點數、退款、補償、機台維修、個資等高風險內容做未經確認的承諾。ECOCO AI 客服應該先把 Rule 與知識庫品質做好，再評估是否導入更多模型。
