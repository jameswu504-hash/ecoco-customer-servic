## 1. 背景

ECOCO 客服知識來源分散於官網 FAQ、Excel FAQ、Meta AI 指令、舊 CommandCenter 專案、客服回覆素材、品牌規範與內部 SOP。資料分散會造成以下問題：

- 客服回覆口徑不一致。
- 新人難以知道哪份資料是最新版。
- AI 若直接讀未整理資料，容易使用過期、重複或未確認內容。
- 知識更新沒有版本紀錄，交接困難。

本專案目標是建立一套可實際運作、可維護、可交接的 AI 客服系統，讓使用者可即時提問，內部人員可持續維護知識庫並追蹤 AI 無法回答的問題。

## 2. 產品目標

| 目標 | 說明 |
| --- | --- |
| 提升常見問題回覆效率 | 讓使用者可即時取得 ECOCO 基礎客服回答 |
| 統一客服口徑 | 以整理後的知識庫作為 AI 回答依據 |
| 控制高風險回答 | 避免 AI 對補點、退款、客訴、帳號等問題做未確認承諾 |
| 建立知識維護流程 | 讓客服與維護者能新增、修改、封存與備份知識 |
| 支援營運觀察 | 提供客服量、問題分類、評分與知識缺口報表 |
| 支援交接 | 以文件、Git JSON 與 PostgreSQL 結構保留可交接資料 |

## 4. 產品範圍

### 4.1 前台 AI 客服

- 提供 ECOCO 品牌化客服頁面。
- 顯示常見問題快捷入口。
- 使用者可輸入問題並取得 AI 回覆。
- 回覆後可給予「有幫助」或「需改善」評分。
- 前台不得顯示內部 RAG chunks、資料庫細節或技術資訊。

### 4.2 AI 回覆流程

- 使用 Claude Sonnet 作為主要回答模型。
- 回答前先檢索 PostgreSQL 中的 ECOCO 知識庫。
- 優先使用 pgvector embedding 語意檢索。
- 若 OpenAI embedding 或 pgvector 不可用，系統退回關鍵字檢索。
- 回覆需遵守 ECOCO 品牌語氣、客服邊界與風險規則。

### 4.3 後台管理

- 查看客服統計、對話紀錄與使用者評分。
- 查看 AI 無法確定回答的知識缺口。
- 新增、修改、封存與恢復知識分類。
- 搜尋既有分類，避免重複建立相似分類。
- 下載目前 PostgreSQL 知識庫為 JSON。
- 產出週報、月報或營運摘要。

### 4.4 知識庫版本管理

系統採用兩層知識管理：

| 層級 | 位置 | 用途 |
| --- | --- | --- |
| 執行層 | PostgreSQL `knowledge_sections` | AI 線上實際讀取的知識 |
| 版本層 | `data/ecoco-knowledge-import.json` | 可回復、可交接、可重新匯入的正式知識包 |

後台修改會立即寫入 PostgreSQL，但不會自動更新 GitHub。重大更新或交接前，需下載 JSON，人工確認後放回 Git。

## 5. 回覆風險控管

AI 不得在未確認情況下承諾以下事項：

- 補點、退款、補償、賠償
- 已完成處理、已派員、已通知工程師
- 會員帳號狀態、個資查詢結果
- 客訴責任歸屬
- 尚未確認的新活動、新站點或合作資訊

高風險問題應遵守：

1. 表達理解與感謝。
2. 說明可能原因或一般規則。
3. 收集必要查詢資訊，例如註冊手機末三碼、回收時間、站點、截圖。
4. 引導客服表單或人工協助。
5. 不做最終判定或補償承諾。

## 6. 資料流

```text
使用者提問
  -> 前台送出 /api/chat
  -> 後端建立 session 並驗證輸入
  -> RAG 檢索 PostgreSQL knowledge_chunks
  -> 讀取 risk_level 與 response policies
  -> 組成 Claude system prompt
  -> Claude 產生回答
  -> 回覆使用者
  -> 儲存 conversations
  -> 使用者評分寫入 ratings
  -> AI 不確定時寫入 unanswered_questions
```

## 7. 資料表與檔案

| 名稱 | 用途 |
| --- | --- |
| `knowledge_sections` | 線上知識庫分類與內容 |
| `knowledge_chunks` | RAG 檢索用片段，自動產生 |
| `conversations` | 使用者與 AI 對話紀錄 |
| `ratings` | 使用者評分 |
| `unanswered_questions` | AI 不確定或需人工補資料的問題 |
| `data/ecoco-knowledge-import.json` | 正式知識包 |
| `data/ecoco-ai-customer-service-database.json` | 知識底稿與來源紀錄 |
| `data/ecoco-response-policies.json` | 回覆政策與風險規則 |

## 8. 非目標

現階段不包含：

- 不直接串接 Zendesk、Gmail 或正式客服工單系統。
- 不讓 AI 直接修改會員點數、優惠券或帳號資料。
- 不讓 AI 自行判定退款、補點或賠償。
- 不將舊 CommandCenter 整包搬進正式系統。
- 不使用本地模型直接回覆真實客戶。
- 不把內部個案處理結果公開給 AI 當一般知識。

## 9. 成功指標

| 指標 | 觀察方式 |
| --- | --- |
| 常見問題可回答率 | 已知 FAQ 是否能穩定得到正確回答 |
| 知識缺口數 | AI 無法確定回答的問題是否逐步下降 |
| 使用者評分 | 「有幫助」與「需改善」比例 |
| 高風險合規 | AI 是否避免未確認承諾 |
| 知識維護效率 | 從發現缺口到補入知識庫的時間 |
| 版本完整性 | PostgreSQL 重大更新是否定期匯出並回寫 Git |

成本效益、節省人力或創造效益等數字，需使用經主管確認的估算公式，不應由系統自動推論。

## 12. 風險與限制

| 風險 | 影響 | 控制方式 |
| --- | --- | --- |
| OpenAI 額度不足 | 語意檢索失效，退回關鍵字檢索 | 監控 Render logs 與 `/healthz` |
| 後台改資料未回寫 Git | 重建環境或交接時資料落差 | 大改版與交接前下載 JSON |
| 知識來源過期或衝突 | AI 回答不一致 | 定期稽核知識品質 |
| 使用者輸入個資 | 法遵與隱私風險 | 遮罩、保存期限、避免提交到 Git |
| AI 過度承諾 | 客訴與品牌風險 | 高風險 Rule、客服抽查與知識修正 |

## 13. 管理摘要

ECOCO AI 客服是一套可維護的客服自動化系統，不只是聊天頁面。它以 PostgreSQL 管理線上知識庫，透過 RAG 讓 AI 依據 ECOCO 資料回答，並用 Rule 控制高風險問題的回答邊界。後台提供知識維護、對話查看、評分、知識缺口與營運報表，使客服資料能持續更新、追蹤與交接。
