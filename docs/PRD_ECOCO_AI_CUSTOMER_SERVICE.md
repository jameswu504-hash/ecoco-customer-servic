# ECOCO AI 客服系統 PRD

## 1. 文件目的

本 PRD 說明 ECOCO AI 客服系統的產品目標、使用者角色、核心功能、資料流、風險控管與後續規劃。此文件提供給主管、客服人員、維護者與工程協作者理解系統範圍與交付內容。

## 2. 專案背景

ECOCO 客服資料來源包含官網 FAQ、Excel FAQ、Meta AI 指令、舊 CommandCenter 專案、客服回覆素材、品牌規範與內部 SOP。這些資料原本分散在不同檔案與系統中，不利於維護，也容易造成回答不一致。

本專案的目標是建立一套可實際運作的 AI 客服系統，讓使用者能即時詢問 ECOCO 相關問題，並讓內部人員可以持續維護 AI 知識庫、追蹤知識缺口與檢查回答品質。

## 3. 產品目標

- 提供符合 ECOCO 品牌語氣的 AI 客服前台。
- 使用 PostgreSQL 管理線上知識庫、對話紀錄、評分與知識缺口。
- 使用 RAG 讓 AI 回覆依據 ECOCO 知識庫，而不是只靠模型記憶。
- 使用 Rule 與風險控管，避免 AI 對點數、退款、補償、帳號與機台異常做未確認承諾。
- 提供後台讓內部人員維護知識、查看對話、處理知識缺口。
- 讓 Git JSON 成為正式版本備份與交接資料來源。

## 4. 使用者角色

| 角色 | 需求 |
| --- | --- |
| 一般使用者 | 快速詢問 ECOCO 服務、點數、站點、回收規則、App 使用問題 |
| 客服人員 | 查看 AI 回覆、處理知識缺口、補充常見問題 |
| 維護者 | 管理知識庫、匯出 JSON、檢查資料品質與部署狀態 |
| 主管 / 總經理室 | 查看系統用途、營運報表、客服議題分類與改善成效 |
| 工程協作者 | 維護程式、資料庫、部署設定與 API 串接 |

## 5. 核心功能

### 5.1 客服前台

- 顯示 ECOCO 品牌化 AI 客服頁面。
- 提供常見問題快捷按鈕。
- 使用者可輸入問題並取得 AI 回覆。
- 回覆後可按「有幫助」或「需改善」。
- 不顯示內部 RAG chunk 或技術資訊給使用者。

### 5.2 AI 回覆

- 主要模型使用 Claude Sonnet。
- 回覆前先檢索 PostgreSQL 中的 ECOCO 知識庫。
- 優先使用 pgvector 語意檢索；若未設定 embedding 或 pgvector 不可用，則退回關鍵字檢索。
- 回覆需遵守 ECOCO 語氣、稱呼與禁止承諾規則。

### 5.3 Rule 與風險控管

- 以 `knowledge_chunks.risk_level` 作為高風險判斷依據。
- 高風險問題需使用保守話術。
- 不可直接承諾補點、退款、已處理、工程師已到場或人工已審核。
- 遇到帳號、點數、優惠券、機台異常等問題，需收集必要資訊並引導客服表單或人工處理。

### 5.4 後台管理

- 查看客服總覽、對話紀錄、評分、知識缺口。
- 新增、修改、封存知識分類。
- 搜尋知識分類，避免重複建立相似分類。
- 下載目前 PostgreSQL 知識庫為 JSON。
- 查看營運報表，包含訊息量、分類、待處理議題與改善紀錄。

### 5.5 知識缺口

- 當 AI 沒有足夠資料時，系統會建立知識缺口。
- 後台人員可查看 AI 原始回覆與問題內容。
- 處理完後可刪除或封存待辦，避免列表無限變長。
- 補上的答案應寫回知識庫。

### 5.6 知識庫版本管理

- PostgreSQL 是線上運作來源。
- `data/ecoco-knowledge-import.json` 是正式版本備份與交接來源。
- 大改版或交接前，需從後台下載 JSON，人工確認後覆蓋回 Git 檔案並 commit / push。

## 6. 資料表與資料來源

| 資料表 / 檔案 | 用途 |
| --- | --- |
| `knowledge_sections` | 線上知識庫分類與內容 |
| `knowledge_chunks` | RAG 檢索用的知識片段 |
| `conversations` | 使用者對話紀錄 |
| `ratings` | 使用者對 AI 回覆的評分 |
| `unanswered_questions` | AI 無法確定回答的知識缺口 |
| `data/ecoco-knowledge-import.json` | 正式知識庫 JSON 備份 |
| `data/ecoco-ai-customer-service-database.json` | 原始整合資料與來源資料 |
| `data/ecoco-response-policies.json` | 回答政策與 Rule |

## 7. 系統流程

```text
使用者提問
→ 前台送出 /api/chat
→ 後端查詢 PostgreSQL knowledge_chunks
→ RAG 取得相關知識
→ 套用 Rule 與 risk_level
→ Claude Sonnet 產生回答
→ 回覆使用者
→ 寫入 conversations
→ 使用者評分寫入 ratings
→ 若 AI 不確定，寫入 unanswered_questions
```

## 8. 非目標

現階段不包含以下範圍：

- 不直接串接 Zendesk、Gmail 或正式客服工單系統。
- 不讓 AI 直接修改會員點數、優惠券或帳號資料。
- 不讓 AI 自行判定退款、補點或賠償。
- 不把舊 CommandCenter 整包搬進正式系統。
- 不直接使用 Gemma、Llama 或本地模型回覆真實客戶。

## 9. 成功指標

| 指標 | 說明 |
| --- | --- |
| 可回答率 | 使用者問題能由知識庫支援回答的比例 |
| 知識缺口數 | AI 無法確定回答的問題數量 |
| 滿意度 | 使用者按「有幫助」或「需改善」的比例 |
| 知識維護效率 | 客服人員補知識與封存過期資料的速度 |
| 高風險合規 | AI 是否避免未確認承諾 |
| 資料版本完整性 | PostgreSQL 與 Git JSON 是否定期同步備份 |

## 11. 後續規劃

### P0：確認語意 RAG 是否啟用

Render 需設定 `OPENAI_API_KEY`，否則 pgvector embedding 不會真正啟用，系統會退回關鍵字檢索。

### P1：知識品質維護

- 持續清理重複知識。
- 確認待處理衝突。
- 補齊常見知識缺口。
- 建立固定月度匯出 JSON 流程。

### P2：客服 Flow 圖與交接文件

- 使用 Whimsical 畫出完整客服 Flow。
- 將一般客服操作、維護者操作與工程部署分開說明。

### P3：模型與 Rule 評測

- 建立測試題庫。
- 比較 Claude、Gemma、Llama 等模型在內部測試的表現。
- 先將本地模型用於資料分類與品質檢查，不直接面向客戶。

## 12. 風險與限制

- 如果 Render 未設定 `OPENAI_API_KEY`，語意 RAG 不會啟用。
- 如果後台改了知識但沒有下載 JSON 回 Git，交接或重建環境時可能缺少最新版知識。
- 使用者可能輸入個資，後續需評估資料保存期限與遮罩策略。
- 本系統尚未與正式工單系統串接，因此高風險問題仍需引導客服表單或人工處理。

## 13. 給主管的摘要

ECOCO AI 客服是一套可維護的客服自動化系統，不只是聊天頁面。它透過 PostgreSQL 管理線上知識庫，使用 RAG 讓 AI 依據 ECOCO 知識回答，並透過 Rule 限制高風險問題的回答邊界。後台提供知識維護、對話查看、評分、知識缺口與營運報表，讓客服資料可以持續更新並留下版本紀錄。
