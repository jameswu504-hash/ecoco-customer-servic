# ECOCO AI 客服系統 PRD

## 1. 文件目的

本 PRD 用來定義 ECOCO AI 客服系統的產品目標、使用者、功能範圍、後台維護流程與驗收標準。此系統需支援實際客服情境，並讓 ECOCO 內部人員能維護知識庫、追蹤知識缺口與理解 AI 回覆依據。

## 2. 背景

ECOCO 擁有多種客服資料來源，包含官網 FAQ、Excel FAQ、Meta AI 指令、舊 CommandCenter 專案、客服回覆範本與內部 SOP。資料分散導致 AI 客服若缺乏整理，容易出現回答不一致、資料過期或無法維護的問題。

本專案目標是建立一套可實際運作的 ECOCO AI 客服網站，讓使用者能快速取得回收、點數、站點、App 操作等常見問題解答，也讓內部人員能持續補充知識庫。

## 3. 產品目標

- 提供符合 ECOCO 品牌語氣的 AI 客服前台。
- 使用 PostgreSQL 儲存實際運作資料。
- 使用 RAG 檢索知識片段，降低 AI 憑空回答風險。
- 建立後台，讓內部人員能查看對話、評分、知識庫與知識缺口。
- 讓知識缺口可以被手動補進知識庫，並在處理後刪除。
- 保留 Git JSON 作為正式資料版本管理來源。

## 4. 使用者角色

| 角色 | 說明 | 主要需求 |
| --- | --- | --- |
| 一般使用者 | ECOCO App 或回收機使用者 | 詢問回收、點數、站點、App 操作問題 |
| 客服人員 | ECOCO 內部維護者 | 查看 AI 回覆品質、補知識庫、刪除已處理缺口 |
| 主管/管理者 | 需要了解系統狀態的人 | 查看資料來源、知識庫規模、後續維護方式 |
| 實習生/維護者 | 負責整理資料與更新專案的人 | 更新 Git JSON、維護文件、協助部署 |

## 5. 使用情境

### 5.1 使用者詢問常見問題

使用者進入客服頁面，看到 ECOCO 開場白後輸入問題。AI 會根據 PostgreSQL 知識庫與 RAG 檢索結果回答。

### 5.2 AI 找不到明確答案

若 AI 判斷知識不足，系統會使用保守話術，不承諾補點、退款或已處理，並將問題記錄到 `unanswered_questions`。

### 5.3 後台人員補知識

後台人員查看知識缺口，人工判斷是否需要補資料。若需要，手動到知識庫管理新增或修改分類內容，完成後刪除該筆知識缺口。

### 5.4 正式資料版本整理

若資料已確認為正式內容，維護者應整理回 Git JSON，讓資料來源可追蹤、可版本管理。

## 6. 功能需求

### 6.1 客服前台

- 顯示 ECOCO Logo 與品牌化介面。
- 顯示開場白：「Hi 可可粉」。
- 支援使用者輸入問題。
- 顯示 AI 回覆。
- 支援常見問題快捷按鈕。
- 支援評分回覆品質。

### 6.2 AI 回覆

- 使用 Claude 產生客服回覆。
- 回覆需符合 ECOCO 語氣：親切、專業、活潑。
- 高風險問題需保守回答，不直接承諾補償。
- 回答前從 PostgreSQL `knowledge_chunks` 檢索相關知識。
- 無法確認時建立知識缺口。

### 6.3 後台總覽

- 顯示 PostgreSQL 知識分類數量。
- 顯示 Git 匯入分類數量。
- 顯示 RAG 檢索片段數量。
- 顯示整理後知識筆數。
- 顯示待確認衝突與資料來源。

### 6.4 知識庫管理

- 顯示所有知識分類。
- 支援分類名稱搜尋。
- 支援新增分類。
- 支援修改分類名稱與內容。
- 支援刪除分類。
- 畫面上隱藏 `AI客服知識：` 前綴，降低閱讀負擔。

### 6.5 知識缺口列表

- 顯示 AI 無法完整回答的問題。
- 顯示 AI 當時的回覆或原因。
- 支援查看 AI 回覆全文。
- 支援刪除已處理或不需處理的缺口。
- 補資料由後台人員手動到知識庫管理完成。

### 6.6 對話與評分紀錄

- 儲存使用者對話紀錄。
- 支援後台查看每次對話。
- 支援評分紀錄，協助後續優化 AI 回覆品質。

## 7. 資料結構

| 資料表 | 用途 |
| --- | --- |
| `knowledge_sections` | AI 使用的主要知識分類 |
| `knowledge_chunks` | RAG 檢索用知識片段 |
| `conversations` | 使用者對話紀錄 |
| `ratings` | 使用者評分紀錄 |
| `unanswered_questions` | 知識缺口紀錄 |

## 8. 資料維護原則

| 入口 | 用途 |
| --- | --- |
| 後台知識庫管理 | 緊急新增、立即生效、客服人員操作 |
| PostgreSQL | 線上系統實際讀取與寫入的資料庫 |
| Git JSON | 正式資料版本管理與內部交接來源 |
| 文件 docs | 說明系統如何運作、如何維護 |

## 9. 不在本階段範圍

- 不直接串接 Zendesk 或 Gmail 自動回信。
- 不開放一般使用者直接修改知識庫。
- 不讓 AI 自動承諾補點、退款、人工處理完成。
- 不在本階段導入 pgvector embedding RAG。
- 不處理正式會員身份驗證或個資查詢。

## 10. 驗收標準

- 使用者可以進入客服前台並送出問題。
- AI 可以根據 ECOCO 知識庫回答常見問題。
- 後台可以查看對話紀錄、評分與知識缺口。
- 後台可以新增、修改、刪除知識分類。
- 知識缺口可以被查看與刪除。
- PostgreSQL 中存在 `knowledge_sections` 與 `knowledge_chunks`。
- Render 部署後網站可正常使用。
- 內部人員能根據文件理解資料來源與維護流程。

## 11. 後續優化建議

1. 在 AI 回覆旁顯示 RAG 命中的知識來源。
2. 將後台新增內容定期整理回 Git JSON。
3. 建立更完整的客服 SOP 分類。
4. 評估導入 pgvector 與 embeddings。
5. 規劃權限分級，例如客服人員、主管、維護者。
6. 導入正式客服系統 API 串接，例如 Zendesk 或內部系統。
